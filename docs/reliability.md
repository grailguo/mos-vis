# MOS-VIS 可靠性与可运维设计（Reliability Spec）

来源：`SystemDesgin.md` 专题拆分  
范围：容错、降级、恢复、测试、运维升级

## 1. 可靠性目标
- 系统在关键依赖异常（设备、模型、网络）下保持可降级服务。
- 连续运行 72 小时无致命崩溃。
- 故障可观测、可定位、可恢复。

## 2. 容错与恢复策略

### 2.1 音频设备异常
- 捕获 ALSA/采集错误后每 1s 重试，最多 10 次。
- 失败后切换备用设备（`default`）并上报告警。

### 2.2 模型加载失败
- VAD/KWS/ASR/TTS 任一模块加载失败时：
  - 禁用该模块并进入降级模式。
  - 主服务保持运行。
  - 记录结构化错误日志。

### 2.3 WebSocket 断连
- 指数退避：`1s,2s,4s,8s,16s...`，上限 `60s`。
- 待发送队列上限：`10` 条。
- 队列满时丢弃最旧请求并记录告警。

### 2.4 看门狗
- 软件看门狗每 1s 检查心跳。
- 超过 5s 无心跳触发重启流程。

### 2.5 systemd 崩溃恢复
```ini
Restart=always
RestartSec=5
StartLimitBurst=3
StartLimitIntervalSec=60
```

## 3. 降级运行模式
- `FULL`：全部能力开启。
- `RECOGNITION_ONLY`：禁用 KWS，按键唤醒。
- `OFFLINE`：禁用 RAG，仅本地 NLU。

降级触发条件：
- 模型缺失/损坏。
- 控制通道连续失败。
- 网络不可用超过阈值。

## 4. 可观测与诊断
- 关键事件：状态迁移、超时、重连、降级、恢复。
- 内置 `LatencyProbe` 记录阶段级时间戳。
- 命令：`--diagnose`

`--diagnose` 自检项：
- 音频设备枚举
- 模型加载
- NPU 可用性
- WebSocket 连通性

## 5. 测试与验证

### 5.1 单元测试
- GoogleTest + Mock 引擎接口。
- 覆盖率目标 > 80%。

### 5.2 集成测试
- 输入音频 -> 期望 JSON 输出。
- CI 在 QEMU aarch64 或真机运行。

### 5.3 性能测试
- 输出平均耗时、P95/P99、吞吐、CPU/NPU 占用、峰值内存。

### 5.4 压力与故障注入
- 72h 稳定性运行。
- 注入场景：网络中断、设备插拔、模型缺失。
- 验收：可恢复且无致命崩溃。

## 6. 运维与升级

### 6.1 日志
- 建议 JSON 结构化日志。
- 关键字段：`session_id`、`pipeline`、`stage`、`latency_ms`、`error_code`。

### 6.2 OTA 与回滚
- 升级流程：下载 -> 签名校验 -> 原子替换 -> 重启。
- 升级失败自动回滚。
- 升级/回滚结果写事件日志。

## 7. 里程碑
- P0：异常处理、降级模式、看门狗、热插拔恢复、测试基线。
- P1：实测性能基准、上下文交互完善、OTA闭环。
- P2：功耗模型精化与长周期稳定性扩展。
